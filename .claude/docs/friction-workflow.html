Friction Log Workflow

A shared notebook of pain points, escalated cleanly to the ZB backend team

TL;DR — You hit an annoying thing in the ZeroBias SDK/client? Run /friction new "short description". Claude walks you through the rest. Scroll down for the full picture.

The Lifecycle

Every friction-log entry moves through these stages. Most never reach the last one — and that's fine.

draft

One dev logged a pain point

→

confirmed

A second dev hit the same thing

→

task-created

Escalated to a ZB platform task

→

resolved

Fix shipped or decision made

→

promoted

Graduated to a customer-facing doc

Why Bother?

  • You stop being the only person who knows. The next dev finds your entry instead of starting from zero.
  • It escalates cleanly. When two devs hit the same thing, the ZB platform task carries real user context, not just a one-line bug report.
  • It feeds the customer-facing dev experience. Resolved friction becomes guides, patterns, skills, and KB articles for external developers.

The Commands

All run from any project via the /friction slash command.

CommandWhat it does
/friction new [title]Create a new entry. Claude asks follow-up questions if needed.
/friction confirm <slug>Add your experience to an existing entry. Bumps draft → confirmed.
/friction task <slug>Create a ZB platform task linked to the entry.
/friction resolve <slug>Mark as resolved, record what actually shipped.
/friction promote <slug>Mark as graduated to a customer-facing resource.
/friction list [status]Show entries, optionally filtered.
/friction checkPoll ZB task status for all escalated entries.

Useful Flags

FlagValues
--severity / -scritical, high, medium, low
--app / -asme-mart, readiness-center, general
--notify / -nPost update to #zb-dx Slack

Walkthrough: A Real Scenario

You're building SME Mart and hit a pain point with the SDK auth flow. Here's the full arc:

Day 1

You hit the pain

You spend an hour figuring out that zerobias-client silently fails to refresh auth tokens when the network drops. You've got a workaround. You're annoyed.

/friction new "Auth token refresh fails silently when offline" -s high -a sme-mart

Claude asks a few questions, creates friction-log/2026-04-09-auth-token-refresh.md, commits, and pushes.

Status: draft

Day 3

Dan hits the same thing

Dan is building Readiness Center and sees a similar pattern in his logs.

/friction confirm auth-token-refresh

Claude finds your entry, asks Dan about his experience, appends a "Confirmed by dan" section, adds him to subscribers, and bumps status.

Status: confirmed

Day 5

You escalate

Two of you have hit this. You both want it fixed in the client library.

/friction task auth-token-refresh --notify

Claude reads the full entry, uses the ZB MCP to create a feature-request task, links it in the frontmatter (zb-task: TASK-4567), and posts to #zb-dx.

Status: task-created

Week 2

The fix ships

Either /friction check detects the status change, or you see the release notes yourself.

/friction resolve auth-token-refresh

Claude asks what shipped, records it, and posts to #zb-dx — subscribers get @-mentioned.

Status: resolved

Week 3

You write a guide

The fix introduces a new API pattern worth documenting for external devs. You write guides/handling-auth-token-refresh.md, then:

/friction promote auth-token-refresh

Claude updates the frontmatter with the destination.

Status: promoted

Setup

Once, per machine:

git clone git@github.com:zerobias-org/zb-dx.git
cd zb-dx
./install.sh

The installer asks:

  1. Your name — used as author in frontmatter
  2. Global or project-level — global is recommended
  3. Symlink or copy — symlink is recommended (auto-updates when anyone pushes)

Verify with ./install.sh status.

Examples

See friction-log/examples/ for five fictional entries — one at each lifecycle stage. They're the fastest way to see what "done right" looks like.

When in doubt: log it. An entry at draft that never moves forward costs nothing. A missed pain point that the next dev rediscovers costs an hour.
Don't log: genuine bugs in your own code, feature requests unrelated to dev experience, ephemeral complaints without specifics, or things you'll fix in 5 minutes.
